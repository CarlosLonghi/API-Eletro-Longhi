# 📚 GLOSSARY — Domain Terminology

This glossary defines key concepts in the Eletro Longhi system. Understand these terms to accurately implement features and maintain architectural coherence.

---

## Core Entities

### **User**
- **What**: A person with authentication credentials who operates the system.
- **Key Fields**: `id`, `userName` (unique), `email`, `password`, `createdAt`, `updatedAt`
- **Role**: Authentication entity; all entities are associated with a user context via JWT claims.
- **Key Pattern**: Users are created via `POST /auth/register` and authenticated via `POST /auth/login`.
- **Example**: Carlos (technician) registers → receives JWT → accesses protected endpoints.

### **Brand** (Marca)
- **What**: A manufacturer of devices (e.g., Samsung, LG, Sony).
- **Key Fields**: `id`, `name` (unique), `createdAt`, `updatedAt`
- **Relationships**: One Brand → Many Devices.
- **Key Pattern**: Devices are filtered by brand; deletion is restricted if devices reference the brand (409 conflict).
- **Example**: Samsung; used to organize device catalog.

### **Device** (Aparelho)
- **What**: An electronic device brought in for repair (e.g., Samsung TV, LG Refrigerator).
- **Key Fields**: `id`, `model`, `serialNumber`, `brand` (FK), `accessories` (many-to-many), `createdAt`, `updatedAt`
- **Relationships**: Many-to-One Brand; Many-to-Many Accessories; One-to-One RepairOrder.
- **Key Pattern**: Devices can have multiple accessories; deletion is restricted if a repair order references it.
- **Example**: A Sony TV (model KDL-55XE9005) with model number and S/N; associated with brand "Sony".

### **Accessory** (Acessório)
- **What**: A component or spare part associated with a device repair (e.g., remote control, power cable, panel).
- **Key Fields**: `id`, `name` (unique), `price`, `createdAt`, `updatedAt`
- **Relationships**: Many-to-Many Devices.
- **Key Pattern**: Accessories are a lookup table; multiple devices can use the same accessory; deletion restricted if in use.
- **Example**: "Remote Control" (price 25.00); shared across many device repairs.

### **Customer** (Cliente)
- **What**: A person who owns the device and brings it for repair.
- **Key Fields**: `id`, `name`, `email`, `phone`, `cpf`, `createdAt`, `updatedAt`
- **Relationships**: One Customer → Many RepairOrders (implicit; orders contain customer contact).
- **Key Pattern**: Customer listing supports advanced filtering + pagination (`name`, `email`, `phone`, `page`, `size`, `sortBy`, `direction`).
- **Example**: João Silva; brought his Sony TV for repair on 2025-07-14.

### **RepairOrder** (Ordem de Reparo)
- **What**: A service ticket that tracks a device repair from intake to completion.
- **Key Fields**: `id`, `device` (FK, unique), `customer` (embedded: name, email, phone, cpf), `status` (enum), `description`, `diagnosis`, `cost`, `createdAt`, `updatedAt`
- **Relationships**: One RepairOrder ↔ One Device (unique constraint); Many RepairOrders ← One Brand (transitive via device).
- **Status Enum**: `OPEN` | `IN_PROGRESS` | `COMPLETED` | `CANCELED`
- **Key Pattern**: Each device has at most one active repair order; order status tracks workflow.
- **Example**: Repair order #12345 for Sony TV (IN_PROGRESS) → technician runs diagnostics → status → COMPLETED.

## Architectural Concepts

### **JWT (JSON Web Token)**
- **What**: Stateless authentication token issued after login.
- **Token Lifetime**: Default 24 hours (configurable).
- **Claims**: `userId`, `userName`, `iat`, `exp`
- **Usage**: Sent via `Authorization: Bearer <token>` header on all protected endpoints.
- **Key Pattern**: Generated by `config/TokenService.generateToken`; verified by `config/SecurityFilter`.
- **Example**: User logs in → receives token → includes token in header for subsequent requests.

### **Spring Boot Layered Architecture**
- **Controller Layer** — Accepts HTTP requests, validates DTOs, delegates to Service.
- **Service Layer** — Implements business rules, coordinates repositories, returns entities/collections.
- **Repository Layer** — Queries/mutations via Spring Data JPA; translates to SQL (PostgreSQL).
- **Entity Layer** — JPA-annotated POJOs representing database rows.
- **Mapper Layer** — Converts between DTOs (request/response records) and Entities; powered by MapStruct.
- **Key Pattern**: Request → Controller → Mapper.toEntity → Service → Repository → Entity → DB; response flows backward.

### **DTO (Data Transfer Object)**
- **What**: Record-based immutable objects for HTTP request/response serialization.
- **Request DTOs**: Located in `controller/request/*.java`; validated with `@Valid` and `@NotNull` / `@NotBlank` annotations.
- **Response DTOs**: Located in `controller/response/*.java`; returned as JSON.
- **Key Pattern**: Never expose entities directly; always map entity → response DTO.
- **Example**: `DeviceRequest` (input) → Device (entity) → `DeviceResponse` (output).

### **Mapper (MapStruct)**
- **What**: Annotation-processor-based code generator for converting between entities and DTOs.
- **Convention**: 
  - `toEntity(RequestRecord) → Entity` — Maps request DTO to entity for persistence.
  - `toResponse(Entity) → ResponseRecord` — Maps entity to response DTO for HTTP output.
- **Helper Methods**: Custom default methods (e.g., `brandFromId(Long id) → Brand`) for ID-to-entity resolution.
- **Key Pattern**: Mappers are Spring beans (`@Mapper(componentModel = "spring")`); inject and call mapper methods.
- **Example**: `DeviceMapper.toEntity(deviceRequest)` generates a Device entity with brand lookup.

### **Flyway Migration**
- **What**: Version-controlled SQL migration script for database schema evolution.
- **Location**: `src/main/resources/db/migration/V*.sql` (e.g., V1, V2, …, V7).
- **Execution**: Automatic on app startup; tracked in `flyway_schema_history` table.
- **Key Pattern**: Never modify old migrations; add new ones; always backward-compatible.
- **Example**: V1 creates `brands`; V2 creates `accessories`; V3 creates `devices` with FK to brands.

### **Security Filter**
- **What**: HTTP request interceptor that extracts JWT claims and populates `SecurityContext`.
- **Location**: `config/SecurityFilter.java`
- **Extraction**: Reads `Authorization: Bearer <token>` header; verifies token signature; extracts `userId`, `userName`.
- **Key Pattern**: Stateless; no session state; each request independently authenticated.
- **Example**: Request arrives → filter extracts JWT → validates signature → sets `JWTUserData` → request continues to controller.

### **REST Endpoint**
- **What**: HTTP method + path combination accepting DTOs and returning responses.
- **Verbs**: GET (retrieve), POST (create), PUT (update), DELETE (remove).
- **Status Codes**: 
  - 200 OK (read/update success), 201 Created (create success)
  - 400 Bad Request (validation failure), 403 Forbidden (auth failure)
  - 404 Not Found (resource missing), 409 Conflict (constraint violation)
- **Key Pattern**: Public endpoints (e.g., `/auth/register`, `/auth/login`); protected endpoints require JWT.
- **Example**: `GET /device/{id}` → returns 200 + Device or 404 + error.

### **Pagination (`Page<T>`)**
- **What**: Paginated response model from Spring Data containing `content` plus metadata (`number`, `size`, `totalElements`, `totalPages`, etc.).
- **Where used**:
  - `GET /device` and `GET /device/search`
  - `GET /customer`
  - `GET /repair-order`
- **Shared params**: `page`, `size`, `sortBy`, `direction`.
- **Utility**: `controller/support/PaginationUtils.createPageable(...)` centralizes `PageRequest` + sort direction parsing.

### **Specification (Spring Data JPA)**
- **What**: Dynamic query composition via predicates (`Specification<T>`) without hardcoding many repository methods.
- **Where used**:
  - `repository/specification/DeviceSpecification.java`
  - `repository/specification/CustomerSpecification.java`
  - `repository/specification/RepairOrderSpecification.java`
- **Key Pattern**: Optional filters add predicates only when request params are present.

## Invariants & Constraints

### **Architectural Invariants**
1. **JWT is the only authentication mechanism** — No sessions; all requests must include a valid bearer token (except `/auth/*`).
2. **Controllers never directly expose entities** — Always map entity → response DTO.
3. **Services return Optional for single-entity lookups** — Controllers use `ifPresentOrElse` to decide 200 vs 404.
4. **Repositories use Spring Data naming conventions** — `findDevicesByBrandId(Long)` instead of custom `@Query`.
5. **Validation happens at controller boundary** — `@Valid` on request DTOs; global exception handler translates to error response.

### **Database Constraints**
1. **Brand.name is unique** — Prevents duplicate manufacturers.
2. **Accessory.name is unique** — Prevents duplicate spare parts.
3. **Device.serialNumber is unique** — Prevents duplicate device records.
4. **RepairOrder.device_id is unique** — At most one active repair order per device.
5. **Foreign keys cascade on delete where appropriate** — Deleting a brand may fail if devices reference it (409).

### **API Contracts**
1. **Deletion returns 204 No Content** — Not 200 OK; no response body.
2. **Update returns 200 OK + updated entity** — Not 204; client sees applied changes.
3. **Creation returns 201 Created + new entity** — Client receives generated ID.
4. **List endpoints may return empty arrays** — Never null; always 200 + [].
5. **Listing profile split is intentional**:
   - `Brand` and `Accessory` remain plain `List` endpoints (no pagination/filter).
   - `Device`, `Customer`, and `RepairOrder` return `Page` and accept filters.

## Workflow Example: Create Repair Order

1. **Technician logs in**: `POST /auth/login` → receives JWT.
2. **Technician looks up device**: `GET /device/{deviceId}` → sees brand, model, accessories.
3. **Technician creates repair order**: `POST /repair-order` with `RepairOrderRequest` (device ID, customer name/email/phone, description, status=OPEN) → receives `RepairOrderResponse` with ID.
4. **Order progresses**: `PUT /repair-order/{id}` updates status → OPEN → IN_PROGRESS → COMPLETED.
5. **Technician closes order**: Status = COMPLETED; order is archived.
